This document describes uses cases and requirements for orientation locking.

Introduction

The combination of portable devices, touch screens (the hardware no longer defines top from bottom), and accelerometers makes it possible to design devices whose screen automatically rotates so that it is always horizontal. This lets the user pick whichever orientation he prefers and lets the viewport automatically adapt. Some developers have started using this to provide two different views of their application. For example, one that's centered around creating new content, and the other, more around bulk editing existing content.

While autorotation is great for most use cases, there are a number of drawbacks. First and foremost, it assumes the user is standing or sitting up, as the viewport's orientation is decided by an accelerometer. Using such a device while lying on the side won't work: the content will always appear perpendicular to the viewer's eyes however it is rotated. This is why such devices are usually equipped with a system-level orientation lock.

Another important drawback is that the automatic rotation of the screen may disrupt use flow. This is particularly true of games that use the accelerometer as a controller.

Finally, autorotation also forces the developer to design his application with these two completely different display formats in mind. For some applications, this may be a significant burden for the developer or might yield a subpar experience for the user.

This document aims to collect use cases and requirements in order to help determine what the best solution to these issues is.
